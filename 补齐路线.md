# 识海 — 补齐路线

> 18 个已知问题，按依赖关系和影响范围排成四个阶段。
> 每个阶段有明确的"完成标准"——不是做完了，是能跑了。

---

## Phase 0：能让它跑起来

> **不解决这两个，代码一行都不用写。**

### #7 硬件装不下文档列的专家数量

**现状**: RTX 3060 12GB 显存，一个 7B FP16 模型 ~14GB——装不下。六个专家全列在文档里，但物理上最多并行 2 个 4bit 量化模型。

**方案**: 分两阶段——先不要专家，再精准加回。

#### Phase 0：无专家运行

```
一次查询：
  分词 → 匹配种子 → 涟漪传播 → 激活分布

"回答" = 当前激活的种子 + 传播路径：
  {
    激活种子: ["感冒", "发热", "咳嗽", "维C", "姜汤"],
    传播路径: [
      "感冒" → COOCCURS → "发热" (权重 0.72),
      "感冒" → TREATED_BY → "维C" (权重 0.68),
      "发热" → RELATED → "咳嗽" (权重 0.55)
    ]
  }
```

不经过任何 LLM 专家。直接返回激活种子列表和传播路径。

**为什么先这样做**: Phase 0 的目标是验证架构——路由对不对、涟漪传播对不对、校验器对不对。这些和专家模型的质量完全无关。用一个黑盒专家回答反而会掩盖路由问题。激活种子列表本身比自然语言回答更透明——能直接看到系统"在想什么"。

**显存**: 0。知识库 SQLite 在内存中 <2GB。纯 CPU 可运行。

**Phase 0 的检索式回答策略**：

```
不是返回裸 JSON。从激活种子和子图拼装一个可读的回答：

1. 激活种子按 activation 降序排列
2. 取 Top-5 + 它们之间的路径
3. 如果有 CC-CEDICT 释义 → 附上
4. 输出格式:

   "关于「感冒」:
    关联概念: 发热、咳嗽、呼吸道疾病、维C、姜汤
    「感冒」COOCCURS_WITH「发热」—— 发热是感冒常见症状
    「感冒」IS_A「呼吸道疾病」—— 感冒属于呼吸道疾病
    「维C」TREATED_BY「感冒」—— 维C常用于感冒辅助治疗"
```

不是自然语言，但有可读的结构。

#### Phase 1 升级：单模型 + LoRA 切换

Phase 0 架构验过之后，加上自然语言回答：

```
一个 7B Q4 基座模型常驻显存（~5GB）
  医学 LoRA  (~50MB)
  法律 LoRA  (~50MB)
  诗词 LoRA  (~50MB)
  物理 LoRA  (~50MB)
  数学 LoRA  (~50MB)
  常识 LoRA  (~50MB)

路由器选定领域 → 卸载当前 LoRA → 加载目标 LoRA（<0.1s）→ 推理
总显存: ~5.3GB。切换几乎无延迟。
```

不卸载模型本体。LoRA 切换是纯显存内操作——卸载旧适配器权重、挂载新权重，不需要重新加载几十亿参数。

**Phase 1 的混合回答策略**：

```
检索式回答（Phase 0 的路径） +
  ↓
激活种子列表 + 传播路径 → 作为 context 注入专家模型
  ↓
专家模型生成自然语言回答
  ↓
校验器检查回答中的关键词是否在激活区域内
```

专家不是凭空回答——它被激活种子列表和传播路径约束着。这个约束本身就抑制幻觉。

#### 为什么不走 API 路线

不用 DeepSeek API、不用 Ollama HTTP。识海的核心立场之一就是零外部依赖。Phase 0 的无专家方案证明了：**架构不依赖任何 LLM 也能跑**。

#### 完成标准

Phase 0: 一次查询返回激活种子列表 + 传播路径，不经过任何模型。10 个测试查询全部路由正确。

Phase 1: 单模型 + LoRA 切换，自然语言回答 + 校验器闭环。同领域连续查询时回答延迟 < 3s。

### #1 冷启动：初始节点和边从哪来

**现状**: 文档说"导入知识库（复用现有语料）"，没说什么语料、怎么变节点和边。

**可用数据**: 龙珠四代积累的全部知识：

```
概念图 (SQLite, 4.5GB)         → 21,097,578 条三元组, 20 种关系
CC-CEDICT (19MB JSON)          → 121,230 条中英词条 + 释义 + 拼音
成语库 (462KB JSON)             → 29,514 条四字成语
语料库 (541MB TXT)              → 4,559,886 行百科/词典释义
Wikipedia (zhwiki.db, 4GB)     → 430,000 篇中文条目全文
同义词 (概念图 SYNONYM)          → 288,486 条同义/俗称关系
```

**方案**: 四层混合导入——每层从最合适的数据源取，每层解决一种需要。

#### 第一层：骨架（IS_A 层级 + SYNONYM 别名）

**数据源**: 概念图 IS_A (713 万) + SYNONYM (28.8 万)

层级关系是涟漪传播的基础。没有 IS_A，"感冒"激活后"疾病"不会被波及——泛化能力为零。

**SYNONYM 直接解决 Phase 1 #2（文本匹配脆）**: 28.8 万条同义关系不进边表——转为节点的 aliases：

```
"感冒" SYNONYM "伤风"    → "感冒".aliases += ["伤风"]
"计算机" SYNONYM "电脑"   → "计算机".aliases += ["电脑"]
"非典" SYNONYM "SARS"    → "非典".aliases += ["SARS"]
```

IS_A 边初始权重 0.8（层级天然强相关）。SYNONYM 不进边——直接扩展别名。

**预期**: ~300K 节点 + ~700K IS_A 边

#### 第二层：血肉（横向关联）

**数据源**: 概念图 RELATED (85.9 万) + CAUSE (11.5 万) + PART_OF (0.95 万) + COOCCURS_WITH (15.1 万)

横向关联让涟漪不只沿层级传播。"感冒"和"发热"不是父子——COOCCURS_WITH 填上了这种关系。

**权重归一化映射**:

```
概念图置信度 c（龙珠权重语义）→ 识海初始权重（涟漪传播语义）:

  c ≥ 0.9  → 0.7    （高置信 → 容易激活）
  c ≥ 0.7  → 0.5    （中置信 → 可能激活）
  c ≥ 0.5  → 0.3    （低置信 → 需要更强的上游涟漪才能激活）
  c < 0.5  → 丢弃   （全导入是噪声，2100 万条里 90% 低于 0.5）
```

每条导入的边标注 `source_tag: "loong_cg_import"`。Phase 2 做常数调优时，可以对比龙珠导入的边和熏习自己长出来的边的权重分布——如果系统性偏离，调整归一化映射。

**预期**: ~100K 新节点 + ~300K 横向边

#### 第三层：释义（词典补 meta）

**数据源**: CC-CEDICT (12.1 万) + 成语库 (2.95 万)

前两层已经覆盖 ~400K 节点，但节点只有标题没有释义。第三层给节点补 meta：

```
CC-CEDICT:
  "量子力学" → {definition: "quantum mechanics", pinyin: "liàng zǐ lì xué"}
  → 如果种子已存在 → 补充 definition 字段
  → 如果不存在 → 创建新种子

成语:
  "龙飞凤舞" → 创建种子 + domain:"成语"
```

**预期**: 补充 ~50K 节点的释义 + 创建 ~10K 新种子（概念图漏掉的成语/俗语）

#### 第四层：覆盖（Wikipedia 低频种子）

**数据源**: zhwiki.db 430K 中文条目名

概念图覆盖高频概念，但低频但真实存在的条目（人名、地名、冷门术语）不在里面。从 Wikipedia 条目名提取前两层没有的，作为低权重种子入库。

```
提取 Wikipedia 条目名 → 去重（对比前两层）
  → 不在的 → 创建种子
    domain: 从 Wikipedia 分类推断
    初始 activation_bias: 0.05（低频，等熏习自然升温）
```

**预期**: ~50K 新种子

#### 导入总览

```
第一层 (IS_A + SYNONYM):     300K 节点 + 700K 边    ← 结构
第二层 (RELATED/CAUSE 等):   100K 节点 + 300K 边    ← 血肉
第三层 (CEDICT + 成语):        60K 节点 + 0 边       ← 释义
第四层 (Wikipedia):            50K 节点 + 0 边       ← 覆盖

总计: ~500K 节点 + ~1M 边
      导入时间: < 2 小时（SQLite → SQLite, 批量 INSERT）
      覆盖日常查询 > 95%
      同义词覆盖: 28.8 万条（直接解决 Phase 1 #2）
```

**为什么分层而不是全量导入**:

- 全量 2100 万条有 90% 的边置信度低于 0.5 → 噪音
- IS_A 和 RELATED 在涟漪传播中需要不同的初始权重 → 不能统一赋值
- SYNONYM 应该进 aliases 而不是进边表 → 需要单独处理
- CC-CEDICT 的释义和概念图的边是互补的 → 不该二选一

**完成标准**: 一次 `import_knowledge_base.py` 跑完，SQLite 里有 50 万节点 100 万边。10 个测试查询（"感冒了吃什么" / "量子纠缠" / "龙飞凤舞"）全部命中对应的种子。

## Phase 1：让路由基本正确

> **Phase 0 能跑，但路由经常指错方向。先解决该问题。**

### #2 文本匹配太脆

**方案**: 在节点匹配层加一层**模糊匹配 + 同义词扩展**。

```
查询词 → 
  ├── 精确匹配 label（现在已有）
  ├── 精确匹配 aliases（现在已有）
  ├── 编辑距离 ≤ 1 的模糊匹配（"感冒"→"感昌"）
  └── 同义词/俗称映射表（"着凉"→"感冒"，"上感"→"上呼吸道感染"）

同义词映射表来源:
  龙珠概念图里的 SYNONYM 边 → 提取已知同义关系
  手工补充 500-1000 个高频词的常见别名
```

**不引入向量相似度**——保持确定性，不回到龙珠字场路线。

**完成标准**: 10 个常见查询词（"着凉"/"拉肚子"/"睡不着"/"上火"）全部精确命中正确种子。

### #5 组合词拆散

**方案**: 在分词之后加一个反向合并。

```
"量子力学" 被分词为 ["量子", "力学"]
  → 先分别查：种子"量子"存在、种子"力学"存在
  → 合并查："量子力学"作为整体是否存在？
  → 存在 → 激活"量子力学"，不分别激活"量子"和"力学"
  → 不存在 → 分别激活
```

**完成标准**: "量子力学"、"人工智能"、"深度学习" 在知识库中作为单个种子被激活，不被拆散。

### #6 否定和反问

**方案**: 在分词时识别否定词和反问标记。

```
"不是感冒" → {"否定的": ["感冒"]}
"怎么可能感冒" → {"反诘": ["感冒"]}
"除了感冒还有什么" → {"排除": ["感冒"], "目标": "其他疾病"}
```

被标记为否定的种子不激活，或者激活但传播方向走排除路径。**

**完成标准**: "不是感冒" 不会激活感冒种子及其邻居。

### #9 多专家答案冲突合并

**方案**: 合并策略分三层。

```
两个专家都回答 → 答案比对:

  1. 一致性高（>80% 重叠）→ 合并为单一回答，置信度取均值
  2. 部分矛盾 → 分别呈现，标注各自的置信度
  3. 完全矛盾 → 常识专家仲裁，或以高置信度为准
```

**完成标准**: "感冒吃什么" 给医学+常识两个专家，返回一个合并后的答案而非两条独立消息。

### #16 并发隔离

**方案**: 按 user_id 分 session。

```
每个 session:
  - 独立的激活值数组（不与全局激活值混淆）
  - 查询开始 → 用户种子激活 → 涟漪在 session 内传播
  - 查询结束 → session 激活值清除
  - 熏习写回全局业力（全局业力是所有用户共享的）

实现: 每个查询维护一个局部 activation dict
     全局图节点的 activation 字段改为 session 级别
     或: 图节点不存 activation，路由传参
```

**完成标准**: 两个用户同时查询不同内容，涟漪互不干扰。

### #15 用户识别

**方案**: Phase 1 不做复杂的鉴权。起一个临时映射表。

```
来源平台 + 来源ID → user_id
  微信: o9cq808HSq... → user_lzk
  Web:  cookie_xxx → user_anon_001
  新来源 → 自动创建 USER 类型节点，label 用来源标识
```

**完成标准**: 同一微信用户连续三条消息，全部命中同一个用户种子。

## Phase 2：让学习不再犯错

> **路由基本正确之后，修正学习机制的缺陷。**

### #10 校验器不能检测"对不对"

**方案**: 在知识库匹配之上加**多专家交叉验证**。

```
专家A 说 "感冒可以用阿奇霉素"
专家B 说 "感冒是病毒性的，抗生素无效"

校验器:
  知识库检查 → 两个答案的关键词都在库里 → 无法区分
  交叉验证 → A 和 B 矛盾 → 置信度打折 → 标记为"存疑"
```

这不能完全解决——如果所有专家都错，交叉验证也救不了。但它能抓住"不同专家给了不同答案"的情况。

**完成标准**: 医学专家和常识专家在感冒用药上产生冲突时，系统返回"存疑"标识。

### #11 熏习粒度太粗

**方案**: 不熏所有 co-activated pairs。只熏**共激活度最高的 top-20 对**。

```
一次查询激活了 300 个种子 → 可能的种子对 44,850 对

改为:
  取激活值最高的 20 个种子
  → 它们之间的 C(20,2) = 190 对
  → 只熏这 190 对
  → 减少了 99.6% 的噪声修正
```

**完成标准**: 单次查询的熏习目标对 ≤ 500 对（从 4 万对降下来）。

### #12 业力无上界

**方案**: 裁剪到 [0.01, 2.0]。

```
所有业力边:
  weight = clamp(weight, 0.01, 2.0)
  
  低于 0.01 → 删除边（彻底冷却，后续不再传播）
  高于 2.0  → 裁剪到 2.0（防止某条边完全垄断）
```

**完成标准**: 任何业力边不超过 2.0，低于 0.01 的边 24 小时内自动删除。

### #4 衰减系数/阈值是拍脑袋的常数

**方案**: 不是用 ML 调，是用**统计观测**调。

```
BFS 衰减系数（当前 0.7）:
  在 100 次查询上统计传播深度 vs 最终匹配精确度
  找到使精确度最高的衰减系数
  → 不一定是 0.7，可能是 0.55 或 0.8

领域阈值（当前 0.3）:
  在 100 次查询上统计阈值 vs 误报率/漏报率
  取 F1 最高的值
```

不是"找个最优常数"——是"证明当前常数是最优的或换成更好的"。

**完成标准**: 三个关键常数（衰减系数、领域阈值、正向熏习条件）各有 100 次查询的统计依据。

### #8 专家质量差异没建模

**方案**: 每个专家有 reliability 字段，校验器对低可靠专家的答案自动打折。

```
常识专家（1B）reliability: 0.6
医学专家（7B）reliability: 0.85

校验器:
  实际置信度 = 原始置信度 × reliability
  常识专家的 0.9 置信度 → 0.9 × 0.6 = 0.54 → 堕入"不确定"区
  医学专家的 0.9 置信度 → 0.9 × 0.85 = 0.765 → 保持"可信"
```

**完成标准**: 常识专家的答案置信度自动乘以 0.6 后才进入熏习决策。

## Phase 3：让系统自生长

> **核心链路稳定后，赋予系统自我扩展能力。**

### #3 别名自动扩展

**方案**: 校验器监测到"词 X 每次都被专家答案中的 Y 覆盖"时，自动把 Y 加入 X 的别名列表。

```
用户问 "着凉怎么办"
→ "着凉" 匹配不到种子 → 常识专家回答中反复出现 "感冒"
→ 统计: 10 次查询 "着凉"，8 次专家答案回指 "感冒"
→ 自动: "感冒" 别名列表增加 "着凉"
→ 下一次 "着凉" 直接匹配到 "感冒" 种子
```

**完成标准**: 运行一周后，"着凉"→"感冒"、"上火"→"炎症" 等 50+ 别名自动入库。

### #18 知识库自动增长（候选种子）

**方案**: 校验器发现某个关键词连续出现 N 次但匹配不到种子 → 创建候选种子。

```
用户连续 5 次问含有 "DeepSeek" 的问题
→ "DeepSeek" 匹配不到任何种子
→ 候选种子: {label:"DeepSeek", candidate_since: "...", count: 5}
→ count ≥ 10 → 自动转为正式种子
→ 专家被要求解释 "什么是 DeepSeek"
→ 解释中的关键词（"大模型"、"AI"、"中国"）成为新种子的初始邻接边
```

**完成标准**: 运行一个月后，知识库自动增长了 500+ 新种子。

### #14 新用户冷启动

**方案**: 新用户继承全局知识库的业力平均值，但不继承其他用户的个人业力。

```
新用户 u_new:
  创建用户种子 → 初始激活 → 
  全局知识库业力是共享的（涟漪传播不受影响）
  个人业力从零开始累积
  前 20 次查询: 系统偏向博识性回答（不假设用户已有背景）
  20 次之后: 用户的个人业力已经足够影响涟漪偏向
```

**完成标准**: 新用户第一次问"那个方程"，系统不会假设他指量子力学中的方程。

### #13 回滚机制

**方案**: 不是真正的"回到过去状态"。是**业力快照 + 手动检查点**。

```
每天凌晨自动 checkpoint:
  → 备份当前所有业力边的权重到 checkpoint.json

发现系统学错了 → 
  → 人工触发回滚到昨天的 checkpoint
  → 或: 不全局回滚，只针对"量子纠缠↔超光速"这条边恢复值
```

**完成标准**: 可以在 1 分钟内恢复到最近一个 checkpoint 的业力状态。

### #17 可观测性

**方案**: 一个轻量监控面板。

```
GET /status → {
  total_seeds: 401234,
  total_karma_edges: 2134567,
  hottest_seeds: ["感冒", "量子力学", "薛定谔方程", ...],
  coldest_seeds: ["轱辘", ...],
  heaviest_karma: ["感冒↔发热:1.98", ...],
  recent_queries: 10 条最近查询 + 路由结果 + 置信度,
  alerts: ["'量子纠缠↔超光速' 权重异常高", "'DeepSeek' 候选种子已 9/10"]
}
```
